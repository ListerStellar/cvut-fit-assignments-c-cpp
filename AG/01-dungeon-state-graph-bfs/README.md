# Dungeon plan: implicit graph, 0-1 BFS (`find_shortest_path`)

**Course:** AG (C++).

## Problem as a graph

The dungeon is a **sparse undirected graph** on `RoomId`. The *search graph* is different: vertices are **full game states** (room, inventory up to 3 item slots, flags for “last room combat outcome”, “treasure collected”, derived hero stats). Edges are **actions**: `Move` (to a neighbor room), `Pickup`, `Drop`. Transitions apply combat rules (monster vs hero with stacking atk/def), stealth, and equipment rules from the PDF.

So you have an **implicit graph**: edges are generated by `getPossibleActions`, not stored.

## Algorithm: 0-1 BFS (not vanilla BFS, not Dijkstra)

- Each **Move** costs **1** action in the objective (minimize number of actions).
- **Pickup** / **Drop** cost **0** in that objective (same time slice in the story).

That is a **shortest path on a graph with edge weights in {0,1}**. The standard solution is **0-1 BFS** with a **deque**:

- push weight-0 successors to the **front**,
- push weight-1 successors to the **back**,

equivalent to running BFS on levels of increasing total cost without a heap.

**Why not ordinary BFS?** If all edges had weight 1, BFS would be correct. Here 0-cost edges would break the “all edges equal” invariant of BFS.

**Why not Dijkstra?** Dijkstra with a binary heap works for nonnegative weights and would be `O(E log V)` in the state graph. With only 0/1 weights, 0-1 BFS is `O(V + E)` for the **expanded** subgraph (no heap). Dial’s algorithm (bucket queue) is the same family.

## Data structures

- **`std::deque<State>`** for the frontier.
- **`std::unordered_map<State, pair<State, Action>, Hash>`** for parent pointers (reconstruct path).
- **Custom hash** on `State` (room, optional items, stats fields) so hashing is O(state size).

Visited handling is tied to first discovery via the parent map (first time a state is reached is along a shortest 0-1 path).

## Complexity (informal)

Let `S` be the number of **distinct states** actually enqueued, `E` the number of **valid transitions** explored from them.

- Time: **`O(S + E)`** for 0-1 BFS, plus **`O(1)`** combat simulation per transition (the course supplies an efficient enough combat helper).
- Space: **`O(S)`** for the deque (worst case) plus **`O(S)`** for predecessor storage.

`S` can grow quickly with rooms, items, and stat combinations; the assignment assumes the test data keeps the state space manageable.

## Tech

`std::variant` for `Action`, `std::optional` for monsters/items, `std::unordered_*`, C++20 `std::ranges::find` in helpers.
